# 관리 보안 설정 및 보안 관련 사항

## 보안에 영향을 미치는 관리 역할

Adobe Learning Manager은 RBAC(역할 기반 액세스 제어) 모델을 사용합니다. 다음 표에서는 FedRAMP 계정 계층을 이 문서 전체에서 참조되는 ALM 역할에 매핑합니다.

| FedRAMP 기간 | ALM 역할 | 보안 관련성 |
|-------------|---------|------------------|
| 최상위 관리 계정 | 책임자 | 전체 계정 수준 제어. 이 문서에 설명된 모든 보안 관련 설정에 대한 배타적 액세스. 이는 &#39;최상위 관리 계정&#39;이라는 용어가 사용될 때 이 안내서 전체에서 참조되는 역할입니다. |
| 권한이 부여된 계정(범위) | 사용자 정의 관리자 | 특정 기능, 사용자 그룹 또는 카탈로그에 범위가 지정된 위임된 관리 액세스 권한입니다. 명시적으로 부여하지 않으면 계정 수준 보안 설정에 액세스할 수 없습니다. |
| 권한 있는 계정(통합) | 통합 책임자 | 통합, API 등록 및 커넥터 구성을 관리합니다. 통합 관리에 대한 상위 권한. 다른 계정 수준 보안 설정을 수정할 수 없습니다. |

>[!NOTE]
>
>이러한 역할이 명시적으로 할당된 사용자만 관리 또는 보안 관련 설정을 수정할 수 있습니다. 별도로 명시되지 않는 한 섹션 3-7에 설명된 설정은 관리자 역할만 액세스할 수 있습니다.

## 로그인 및 인증 설정

로그인 및 인증 설정은 관리자를 포함한 모든 사용자가 ALM 플랫폼에 액세스하는 방법을 제어합니다. 이러한 설정은 책임자 역할로만 구성되며 전체 계정의 보안 상태에 직접적이고 중요한 영향을 미칩니다.

### 로그인 방법 구성

**위치:** ALM 관리자 > 설정 > 로그인 방법

책임자는 모든 내부 및 외부 사용자에게 사용되는 인증 방법을 제어합니다. 옵션은 다음과 같습니다.

- **Adobe ID:** 사용자는 개인 Adobe 계정을 통해 인증합니다. 조직이 아닌 Adobe이 관리합니다.
- SAML 2.0을 통한 **SSO(Single Sign-On):** 사용자는 조직의 ID 공급자(IdP)를 통해 인증합니다. 조직은 인증, MFA 및 세션 정책을 완전히 제어합니다.
- **Learning Manager ID:** 외부 사용자만 사용할 수 있습니다. 사용자는 플랫폼별 사용자 이름과 암호를 만듭니다.
- **다중 SSO 구성:** 최대 20개의 SSO 구성을 추가하여 다양한 사용자 그룹, 위치 또는 조직 구분을 지원할 수 있습니다.

| 로그인 방법 | 보안 함축 | 추천 |
|-------------|---------------------|----------------|
| **Adobe ID** | 조직에는 암호 정책, MFA 또는 계정 복구를 제어할 수 없습니다. 손상된 개인 Adobe 계정은 ALM 플랫폼에 대한 액세스 권한을 부여합니다. | 관리자 또는 내부 사용자에게는 **권장되지 않음**. SSO를 사용할 수 없는 경우에만 사용합니다. |
| **SSO(SAML 2.0/Federated ID)** | 인증은 조직의 IdP에 의해 완전히 제어됩니다. MFA, 세션 시간 초과 및 조건부 액세스 정책은 IdP 수준에서 적용됩니다. 사용자 출발에 대한 즉각적인 취소. | 모든 내부 사용자 및 관리자에게 **권장**. 최고 수준의 조직 제어 기능을 제공합니다. |
| **Learning Manager ID** | 조직의 ID 인프라 외부에서 사용자가 직접 암호를 관리합니다. ALM을 통해 MFA를 시행할 수 없습니다. 암호 강도는 사용자 동작에 따라 다릅니다. | 외부 사용자만 사용할 수 있습니다. 직원이나 관리자에게 적합하지 않습니다. |

>[!CAUTION]
>
>로그인 방법이 내부 사용자용 Adobe ID으로 설정된 경우 조직은 다단계 인증을 시행하거나, 암호 복잡성을 제어하거나, 사용자가 나갔을 때 즉시 액세스를 취소할 수 있는 기능을 상실합니다. 이는 무단 액세스의 위험을 크게 증가시킵니다.

자세한 내용은 [사용자 지정 역할](https://experienceleague.adobe.com/ko/docs/learning-manager/using/admin/custom-role)을 참조하세요.

### Multi-Factor Authentication(MFA)

**위치:** Adobe Admin Console > 설정 > 개인 정보 보호 및 보안 > 인증 설정

**Adobe ID** 및 **Enterprise ID** 사용자에 대한 MFA 적용은 ALM 응용 프로그램 자체가 아닌 Adobe Admin Console의 **시스템 관리자**&#x200B;에 의해 구성됩니다.  **Federated ID/SSO** 사용자의 경우 MFA는 조직의 ID 공급자에서 적용됩니다.

- **강제 2FA:** 사용자는 2단계 인증을 비활성화할 수 없습니다. 자격 증명 도용 및 피싱에 대한 강력한 보호 기능을 제공합니다.
- **선택적 2FA:** 사용자가 2단계 인증을 사용할지 여부를 선택합니다. 훨씬 더 약한 보안 태세.
- **MFA가 구성되지 않음:** 암호만 액세스를 보호합니다. 높은 위험, 특히 관리 계정의 경우.

>[!IMPORTANT]
>
>강제 MFA가 없는 관리 계정은 무단 액세스의 위험이 훨씬 높습니다. MFA가 없는 손상된 관리자 자격 증명은 모든 보안 설정을 수정할 수 있는 기능을 포함하여 모든 계정 제어를 부여합니다.

### 세션 수명 및 유휴 시간 초과

**위치:** Adobe Admin Console > 설정 > 개인 정보 보호 및 보안 > 고급 설정

**시스템 관리자**&#x200B;는 조직의 최대 활성 세션 수명 및 유휴 세션 시간 제한을 구성합니다. 이러한 설정은 관리자를 포함한 모든 사용자에게 적용됩니다.

- **최대 세션 수명:** 활동에 관계 없이 정의된 기간 후에 재인증을 강제로 수행합니다. 세션이 손상된 경우 기회 창을 제한합니다.
- **최대 유휴 시간:** 정의된 기간 동안 비활성 상태인 세션을 종료합니다. 공유 디바이스 또는 잠금 해제된 디바이스에서 무인 인증 세션으로부터 보호합니다.

## 역할 할당 및 권한 범위 설정

역할 할당 및 권한 범위 설정은 ALM에서 관리 액세스 권한을 가진 사용자와 이들이 수행할 수 있는 작업을 결정합니다. 이 설정은 플랫폼에서 가장 큰 영향을 미치는 보안 설정 중 하나입니다. 관리자 역할만 관리 역할을 생성, 할당, 수정 또는 취소할 수 있습니다.

### 관리 역할 할당

**위치:** ALM 관리자 > 사용자 > 내부 > 작업 > 역할 할당/역할 제거

**관리자**&#x200B;는 다음 역할을 부여하거나 취소할 수 있습니다.

- **관리자:** 전체 계정 수준 액세스
- **작성자:** 콘텐츠 만들기 액세스
- **사용자 지정 관리자:** 범위 관리 액세스(섹션 4.2 참조)
- **통합 관리자:** 통합 및 API 액세스(섹션 7 참조)

| 설정 / 작업 | 보안 함축 | 권장 수행 |
|---|---|---|
| **관리자 역할 부여** | 전체 계정 제어. 이 역할을 가진 사용자는 인증 재구성 및 다른 관리자의 액세스 권한 취소를 포함하여 이 문서의 모든 설정을 수정할 수 있습니다. | 확인된 비즈니스 요구 사항이 있는 최소 사용자 수만 할당하십시오. 모든 관리자 역할 보유자의 문서화된 목록을 유지 관리합니다. |
| **출발에서 역할을 해지하지 못함** | 중단된 사용자는 관리 기능에 계속 액세스할 수 있습니다. 무단 액세스, 데이터 유출 또는 파괴 행위의 위험 | 사용자의 고용 또는 관리 책임이 변경되면 즉시 역할을 제거합니다. 정기적인 액세스 검토를 수행합니다. |
| **광범위한 역할 초과 할당** | 광범위한 관리 액세스는 손상된 계정의 잠재적인 영향을 증가시키고 관리 작업에 대한 책임을 줄여줍니다. | 최소 특권의 원칙을 적용하라. 가능한 경우 범위가 지정된 역할(예: 사용자 정의 관리자)을 선호합니다. |

### 사용자 정의 관리 역할 구성

**위치:** ALM 관리자 > 사용자 > 사용자 정의 역할 > 역할 만들기

**최소 권한**&#x200B;의 원칙을 적용합니다. **사용자 지정 관리자** 역할을 사용하여 특정 함수를 위임합니다(섹션 4.2 참조).

**관리자**&#x200B;는 정의된 관리 권한의 하위 집합을 부여하는 사용자 지정 역할을 만들 수 있습니다. 사용자 정의 역할은 특정 사용자 그룹, 카탈로그 또는 기능 세트로 범위를 지정할 수 있으며, 이는 위임된 관리 액세스의 범위를 제한합니다.

**사용 가능한 권한 범주는 다음과 같습니다.**

- **계정 권한:** 공지, 게임화, 스킬 및 사용자 관리와 같은 시스템 전체 구성에 대한 액세스 권한.
- **기능 권한(핵심):** 카탈로그, 보고서 및 태그에 대한 액세스 권한.
- **기능 권한(학습 개체):** 강의, 인증, 학습 경로 및 작업 지원에 대한 액세스
- **범위:** 역할을 특정 사용자 그룹 및/또는 카탈로그로 제한합니다.

| 구성 결정 | 보안 함축 | 권장 수행 |
|---|---|---|
| **지나치게 광범위한 사용자 지정 역할 만들기** | 과도한 권한을 가진 사용자 정의 역할은 전체 관리자 역할에 비해 보안 이점을 거의 제공하지 않으며 손상된 계정의 폭발 반경을 증가시킵니다. | 사용자 지정 역할을 특정 기능에 필요한 최소 권한 집합으로 확장합니다. 카탈로그 범위 및 사용자 그룹 범위 역할을 선호합니다. |
| **사용자 지정 관리자에게 설정 액세스 권한 부여** | 설정 액세스 권한이 있는 사용자 정의 관리자는 로그인 방법, 알림 설정 및 기타 계정 수준 구성을 수정할 수 있습니다. | 명시적으로 필요한 경우에만 사용자 정의 역할에 대한 설정 액세스 권한을 부여합니다. 이 권한이 정기적으로 있는 사용자 정의 역할을 감사하십시오. |
| **사용자 지정 역할 할당을 감사하지 않음** | 문서화되지 않았거나 검토되지 않은 사용자 정의 역할은 시간이 지남에 따라 누적되어 권한 크립으로 이어질 수 있습니다. | 사용자 정의 역할 보고서를 정기적으로 다운로드하고(**사용자 > 사용자 정의 역할 > 다운로드**) 모든 활성 사용자 정의 역할 할당을 검토하십시오. |

## 사용자 프로비저닝 및 액세스 관리 설정

사용자 프로비저닝 설정은 사용자가 플랫폼에 추가되는 방식, 활성 상태로 유지되는 기간 및 데이터가 제거되는 시기를 제어합니다. 이러한 설정은 책임자 역할로만 구성되며 데이터 노출 및 액세스 제어에 직접적인 보안 영향을 미칩니다.

| 설정 | 위치 | 보안 함축 | 권장 기본값 |
|---|---|---|---|
| **비활성 사용자 자동 삭제** | ALM 관리자 > 설정 > 일반 | 자동 삭제 없이 비활성 계정이 누적됩니다. 휴면 계정은 모니터링이 중단될 수 있으므로 일반적인 무단 액세스 대상입니다. | 활성화합니다. 조직 정책에 따라 보존 기간(예: 90일 비활성화)을 설정합니다. |
| **외부 사용자 프로필 만료** | ALM 관리자 > 사용자 > 외부 > 프로필 추가 | 만료일이 없는 외부 사용자 프로필은 무기한 액세스할 수 있습니다. 더 이상 액세스 권한이 필요하지 않은 파트너 또는 계약자는 활성 진입점을 유지합니다. | 생성 시 모든 외부 사용자 프로필에 만료 날짜를 설정합니다. |
| **자체 등록 링크 컨트롤** | ALM 관리자 > 사용자 > 내부 > 추가 > 자가 등록 | 자체 등록 링크를 통해 URL을 가진 모든 사용자가 학습자 계정을 만들 수 있습니다. 관리되지 않는 경우 무단 또는 예기치 않은 사용자가 플랫폼 액세스 권한을 얻을 수 있습니다. | 필요한 경우에만 자체 등록 링크를 생성합니다. 사용하지 않는 링크를 즉시 비활성화하거나 해지하십시오. |
| **외부 프로필 일시 중지/다시 시작** | ALM 관리자 > 사용자 > 외부 > 작업 > 일시 중지 | 외부 프로필을 일시 중지하면 신규 사용자는 등록할 수 없지만 이미 등록된 사용자에게는 영향을 미치지 않습니다. 비활성 외부 프로필을 일시 중지하지 않으면 의도하지 않은 등록이 허용될 수 있습니다. | 등록이 더 이상 필요하지 않을 때 외부 프로필을 일시 중지합니다. 영구적으로 비활성화된 프로필을 삭제합니다. |
| **사용자 삭제 및 제거** | ALM 관리자 > 사용자 > 내부 > 작업 > 삭제/사용자 정리 > 제거 | 삭제된 사용자는 비활성화되지만 데이터는 유지됩니다. 지우면 모든 사용자 데이터가 영구적으로 제거됩니다. 삭제된 사용자를 제거하지 않으면 중요한 학습 기록 및 PII가 필요한 보존 기간 이상 보존될 수 있습니다. | 조직 데이터 보존 및 개인정보 처리방침에 따라 사용자 레코드를 제거합니다. 제거는 되돌릴 수 없습니다. |

>[!IMPORTANT]
>
>제거는 되돌릴 수 없는 영구적인 작업입니다. 모든 학습 기록, 등록 데이터 및 사용자 정보가 삭제됩니다. 제거된 사용자가 커넥터 구성에서 참조되는 경우 해당 커넥터는 비활성화됩니다. 제거를 실행하기 전에 신중하게 선택 사항을 확인합니다.

## 보고 및 데이터 액세스 설정

보고 및 데이터 액세스 설정에 따라 플랫폼 데이터를 보고, 생성하고, 내보낼 수 있는 사용자가 결정됩니다. 이러한 설정을 잘못 구성하면 민감한 개인, 조직 또는 규정 준수 관련 정보가 노출될 수 있습니다.

| 설정 | 위치 | 보안 함축 | 권장 기본값 |
|---|---|---|---|
| **사용자 지정 역할에 대한 보고서 액세스 권한 부여** | ALM 관리자 > 사용자 > 사용자 정의 역할 > 기능 권한 > 보고서 | 보고서에는 개인 식별 정보(PII), 강의 완료 데이터, 평가 점수 및 학습자 진행률이 포함될 수 있습니다. 보고 기능에 대한 액세스는 비즈니스 요구가 확인된 사용자로 제한되어야 합니다. | 문서화된 특정 요구 사항이 있는 역할에만 보고서 액세스 권한을 부여합니다. 쓰기 액세스 권한이 필요하지 않은 경우 읽기 전용 액세스 권한을 사용합니다. |
| **전체 제어 및 읽기 전용 보고서 범위** | ALM 관리자 > 사용자 > 사용자 정의 역할 > 계정 요약 보고서 | 계정 요약 보고서에 대한 전체 제어 는 역할 범위에 관계없이 모든 사용자 그룹 및 카탈로그에 걸쳐 사용자 정의 관리자 가시성을 부여합니다. 이로 인해 범위 지정된 관리자 역할의 의도된 범위 밖에 데이터가 노출될 수 있습니다. | 조직 전체의 보고서 가시성이 진정으로 필요한 역할에 대해서만 계정 요약 보고서에 대한 전체 제어 권한을 부여합니다. |
| **xAPI 및 전자 메일 보고서 액세스** | ALM 관리자 > 사용자 > 사용자 정의 역할 | xAPI 보고서 및 이메일 보고서 는 전체 관리자 역할에서만 사용할 수 있습니다. 이러한 보고서에는 상세한 행동 및 커뮤니케이션 데이터가 포함될 수 있습니다. 액세스는 디자인으로 제한됩니다. | 사용자 정의 역할을 통해 xAPI 또는 이메일 보고서 액세스 권한을 위임하려고 하지 마십시오. 이는 관리자 전용 설계 방식입니다. |
| **내보낸 사용자 데이터** | ALM 관리자 > 사용자 > 내부 > 사용자 데이터 내보내기 | 사용자 데이터 내보내기 기능은 모든 내부 사용자 레코드가 포함된 다운로드 가능한 파일을 생성합니다. 이 데이터는 조직의 보안 및 개인정보 처리방침에 따라 처리되어야 합니다. | 내보내기 기능을 승인된 직원으로 제한합니다. 내보낸 데이터를 민감한 데이터로 처리합니다. 승인된 시스템 외부에 저장하거나 전송하지 마십시오. |

## 통합 및 API 액세스 설정

통합 및 API 액세스 설정은 외부 시스템이 Adobe Learning Manager에 연결되는 방식을 제어합니다. 이러한 설정은 ALM 사용자 인터페이스를 넘어 액세스를 확장하며, 잘못 구성되면 플랫폼 데이터 및 기능이 승인되지 않은 시스템에 노출될 수 있습니다.  통합 설정은 통합 책임자 역할에 의해 관리됩니다. 전체 관리자 역할에는 등록된 응용 프로그램을 검토, 승인 및 해지할 수 있는 기능이 유지됩니다.

| 설정 | 위치 | 보안 함축 | 권장 기본값 |
|---|---|---|---|
| **API 응용 프로그램 등록** | 통합 책임자 > 응용 프로그램 > 등록 | 응용 프로그램을 등록하면 ALM 데이터 및 기능에 프로그래밍 방식으로 액세스하는 데 사용할 수 있는 OAuth 2.0 자격 증명(클라이언트 ID 및 시크릿)이 만들어집니다. 지나치게 넓은 OAuth 범위(예: 관리자 역할 읽기/쓰기)는 전체 관리 API 액세스 권한을 부여합니다. | 필요한 최소 OAuth 범위를 부여합니다. 사용하지 않거나 레거시 응용 프로그램 등록을 주기적으로 검토하고 취소합니다. 클라이언트 자격 증명을 중요한 암호로 취급합니다. |
| **OAuth 응용 프로그램 범위** | 통합 책임자 > 응용 프로그램 > 등록 > 범위 | 사용 가능한 OAuth 범위는 학습자 읽기 액세스에서 관리자 읽기/쓰기 액세스까지 다양합니다. 관리자 역할 읽기/쓰기를 사용하면 사용자 역할 및 보안 설정을 포함하여 관리자가 수정할 수 있는 모든 데이터를 응용 프로그램에서 수정할 수 있습니다. | 통합 요구 사항을 충족하는 가장 제한적인 OAuth 범위를 사용합니다. 꼭 필요한 경우가 아니면 관리자 역할의 읽기/쓰기 액세스 권한을 부여하지 마십시오. |
| **커넥터 구성(FTP, Salesforce, HRIS 등)** | 통합 책임자 > 커넥터 | 커넥터를 사용하여 사용자 데이터, 스킬, 강의 완료 등을 자동으로 가져오고 내보낼 수 있습니다. 잘못 구성된 커넥터는 잘못된 사용자 데이터를 가져오거나, 의도하지 않은 역할을 할당하거나, 중요한 데이터를 승인되지 않은 대상으로 내보낼 수 있습니다. | 모든 활성 커넥터 구성을 주기적으로 검토합니다. 데이터 원본 및 대상이 인증되었는지 확인합니다. 더 이상 사용되지 않는 커넥터를 비활성화합니다. |
| **Webhook 구성** | 통합 책임자 > Webhooks | Webhook은 실시간 ALM 이벤트 데이터(등록, 완료, 사용자 변경)를 지정된 외부 URL로 전송합니다. 잘못 구성되거나 손상된 Webhook 엔드포인트는 민감한 학습자 이벤트의 데이터 추출 또는 노출을 초래할 수 있습니다. | 확인된 조직 승인 Webhook URL만 등록합니다. 활성 Webhook 구성을 정기적으로 검토합니다. 비활성 또는 인식되지 않는 Webhook을 즉시 제거합니다. |
| **LTI 통합 구성** | 통합 책임자 > LTI 통합 | LTI 통합은 ALM이 LTI 공급자 또는 소비자 역할을 할 수 있게 하여 외부 LMS 플랫폼이 ALM 과정에 접근할 수 있게 한다. 일단 활성화되면 LTI를 비활성화할 수 없습니다. 노출된 LTI 자격 증명으로 강의 콘텐츠에 대한 무단 액세스를 허용할 수 있습니다. | 확인된 통합 요구 사항이 있는 경우에만 LTI를 활성화하십시오. LTI 자격 증명을 민감한 항목으로 처리합니다. 권한이 있는 LMS 관리자와만 자격 증명을 공유합니다. |
